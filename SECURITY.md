# Security Policy

## 🔒 Supported Versions

This project is private and proprietary. Only the latest version is actively maintained and supported.

---

## 🚨 Reporting a Vulnerability

If you discover a security vulnerability, **DO NOT** open a public issue.

Instead, report it directly to the maintainer:

* Contact: [[zutshiaryan5@gmail.com](mailto:zutshiaryan5@gmail.com)]

Include:

* Description of the vulnerability
* Steps to reproduce
* Potential impact
* Suggested fix (if available)

You can expect a response within a reasonable timeframe.

---

## ❗ Responsible Disclosure

* Do not disclose vulnerabilities publicly without explicit permission
* Do not exploit vulnerabilities beyond what is necessary to demonstrate the issue
* Allow time for the issue to be resolved before any discussion

Failure to follow responsible disclosure may result in legal action.

---

## 🔐 Security Best Practices for Contributors

All contributors must:

* Never commit secrets (API keys, tokens, credentials)
* Use environment variables for sensitive data
* Validate all inputs and sanitize outputs
* Follow least-privilege principles
* Avoid introducing unnecessary dependencies

---

## 🛡️ Confidentiality Notice

This project is **confidential and proprietary**.

* Do not share, distribute, or discuss the codebase outside authorized access
* Do not expose internal architecture or logic publicly
* Any unauthorized disclosure is strictly prohibited

---

## ⚠️ Enforcement

Violations of this policy may result in:

* Immediate removal of access
* Rejection of contributions
* Legal action where applicable

---

## 📌 Additional Notes

Security is not optional. Every contribution must be reviewed with a security-first mindset.
